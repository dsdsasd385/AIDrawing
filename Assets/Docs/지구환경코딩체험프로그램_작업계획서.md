# 지구 환경 코딩 체험 프로그램 개발 작업 계획서

## 1. 프로젝트 개요

### 프로젝트명

지구 환경 코딩 체험 프로그램

### 개발 목적

블록 코딩을 활용하여 어린이 및 일반 체험자가 기후 변화와 환경 보호의 중요성을 쉽고 재미있게 이해할 수 있도록 제작하는 교육용 체험 프로그램이다.

체험자는 블록을 직접 조립하여 프로그램을 설계하고, 자신이 만든 결과가 실시간 애니메이션과 점수로 표현되는 과정을 통해 변수, 순차 실행, 조건문, 반복문, 함수 등의 기본적인 프로그래밍 개념과 환경 보호 활동을 함께 학습한다.

---

## 블록 연결 방식

블록은 퍼즐처럼 결합된다.

예시

```
┌──────┐
│ 시작 │
└─┐────┘
  │
┌─┴────┐
│ 블록 │
└─┐────┘
  │
┌─┴────┐
│ 블록 │
└─┐────┘
  │
┌─┴────┐
│ 종료 │
└──────┘
```

### 블록 모양

블록의 기능을 색상뿐 아니라 모양으로도 직관적으로 구분한다.

예시

* ㄱ 형태 → 시작 블록
* ㅡ 형태 → 일반 명령 블록
* ㄴ 형태 → 변수 블록
* ㄷ 형태 → 조건문 블록
* U 형태 → 반복문 블록
* 캡슐형 → 함수 블록

색상을 구분하지 않아도 형태만 보고 어떤 블록인지 유추할 수 있도록 제작한다.

# 12. 전시장 운영 및 유지보수 설계

## 개발 목표

본 프로그램은 전시장 및 체험관에서 장기간 운영되는 콘텐츠를 목표로 개발한다.

운영 중 발생할 수 있는 콘텐츠 변경, 문구 수정, 점수 조정 등을 개발자가 직접 수정하지 않아도 운영자가 쉽게 변경할 수 있도록 데이터 중심(Data Driven) 구조로 설계한다.

---

# 13. 시스템 구조

프로그램은 크게 다음과 같이 구성한다.

```text
UI

↓

블록 시스템

↓

게임 로직

↓

데이터 관리

↓

JSON / ScriptableObject
```

UI와 게임 로직을 분리하여, 데이터만 변경해도 콘텐츠가 자동으로 반영되는 구조를 적용한다.

# 15. JSON 기반 데이터 관리

운영자가 쉽게 수정할 수 있도록 JSON 파일을 사용한다.

예시

```json
{
    "EpisodeName":"지구 온도 낮추기",
    "Description":"CO₂를 줄여 지구 온도를 낮춰보세요.",
    "Score3":3,
    "Score2":2,
    "Score1":1
}
```

프로그램 실행 시 JSON을 읽어 화면에 자동 적용한다.

JSON만 수정하면 프로그램을 다시 빌드하지 않아도 내용 변경이 가능하도록 설계한다.

# 19. 유지보수 구조

각 기능은 독립적인 Manager로 구성한다.

```text
GameManager

├── UIManager
├── EpisodeManager
├── BlockManager
├── DataManager
├── AnimationManager
├── AudioManager
├── ScoreManager
├── SaveManager
└── SceneManager
```

Manager 간 의존성을 최소화하여 특정 기능 수정 시 다른 시스템에 영향을 주지 않도록 설계한다.

---

# 20. 오류 대응

프로그램 실행 중 오류 발생 시 강제 종료되지 않도록 예외 처리를 적용한다.

예시

* JSON 파일 없음
* 이미지 없음
* 효과음 없음
* 애니메이션 없음

오류 발생 시 기본 데이터(Default Data)를 사용하여 프로그램이 계속 실행되도록 한다.

모든 오류는 로그 파일에 기록하여 유지보수에 활용한다.

# 24. 확장성

현재는 5개의 에피소드를 제공하지만, 향후 에피소드 추가를 고려하여 구조를 설계한다.

신규 에피소드 추가 시 필요한 작업은 다음과 같다.

1. JSON 데이터 추가
2. 리소스 추가
3. 에피소드 프리팹 추가

기존 소스 코드의 수정 없이 새로운 콘텐츠를 확장할 수 있는 구조를 목표로 한다.

# 26. 코드 주석 작성 규칙

유지보수와 인수인계를 고려하여 모든 스크립트에 주석을 작성한다.

## 작성 원칙

* 모든 스크립트는 한국어 주석으로 작성한다.
* 주석은 코드의 "무엇"이 아니라 "왜"와 "역할"을 설명한다.
* 다른 개발자가 처음 봐도 구조를 파악할 수 있도록 작성한다.

## 필수 작성 항목

### 1. 파일(클래스) 상단 주석

모든 클래스 상단에 XML 문서 주석(`/// <summary>`)으로 다음을 작성한다.

* 클래스의 역할
* 어떤 시스템에 속하는지 (예: 코어 매니저, 블록 시스템, 에피소드)
* 다른 클래스와의 관계 (참조/피참조)

### 2. 공개 메서드 주석

`public` 메서드에는 XML 문서 주석으로 다음을 작성한다.

* 메서드의 기능
* 매개변수 의미 (`<param>`)
* 반환값 의미 (`<returns>`)

### 3. 필드 및 프로퍼티 주석

* `public` / `[SerializeField]` 필드에는 용도를 `/// <summary>`로 작성한다.
* Inspector에 노출되는 필드는 운영자·개발자가 값의 의미를 알 수 있도록 작성한다.

### 4. 로직 내부 주석

* 조건 분기, 계산식, 예외 처리 등 의도가 드러나지 않는 코드에는 한 줄 주석(`//`)으로 이유를 설명한다.
* 점수 계산, 판정 기준 등 기획 수치와 연결되는 부분은 기획 근거를 함께 명시한다.
