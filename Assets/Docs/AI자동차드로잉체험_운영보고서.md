# AI 자동차 드로잉 체험 운영 보고서 요약

> 작성일: 2026-07-13  
> 목적: 현재 작동 중인 워크플로우, 구동 최소 사양, 운영 보고 시 확인해야 할 핵심 항목을 한 문서로 정리한다.

---

## 1. 현재 작동 중인 워크플로우

1. 대기 화면에서 사용자가 시작한다.
2. 그리기 화면에서 자동차 스케치를 입력한다.
3. 선 레이어와 색 레이어를 각각 PNG로 저장한다.
4. 로컬 ComfyUI 서버로 생성 요청을 전송한다.
5. 생성 결과 이미지를 Unity 결과 화면에 표시한다.
6. 세션 결과를 로컬에 저장하고, 설정이 준비된 경우 QR 업로드와 갤러리 등록까지 진행한다.
7. 영상 생성은 결과 화면 표시 후 백그라운드에서 시도한다.
8. 영상 생성 실패 또는 시간 초과가 발생해도 이미지 결과는 유지한다.

### 핵심 동작 요약

- 생성은 로컬 ComfyUI 서버 기반으로 동작하므로 기본 체험은 인터넷 없이도 가능하다.
- QR 업로드, 갤러리 등록, 콘텐츠 필터는 부가 기능이며 실패해도 기본 체험은 유지된다.
- 스타일은 실사, 카툰, 픽셀아트, 클레이 4종으로 구성되어 있으며 `Styles.json`에서 관리한다.
- 결과 이미지는 세션별로 로컬 저장된다.

## 2. ComfyUI API 및 생성 워크플로우

### 사용 API

Unity는 ComfyUI를 로컬 HTTP 서버로 실행한 뒤 아래 API를 호출한다.

- `/upload/image`: 사용자가 그린 선 레이어와 색 레이어 PNG를 ComfyUI 입력 이미지로 업로드한다.
- `/prompt`: Unity가 워크플로우 JSON의 일부 노드를 세션별 값으로 치환한 뒤 생성 작업을 제출한다.
- `/history/{prompt_id}`: 제출한 작업의 완료 여부를 0.5초 간격으로 조회한다.
- `/view`: 생성이 완료된 결과 이미지 또는 영상을 다운로드한다.

기본 서버 주소는 `http://127.0.0.1:8188`이며, 현재 이미지 생성 워크플로우 파일은 `Assets/StreamingAssets/ComfyUI/car_workflow_api.json`을 사용한다. 영상 생성은 별도 워크플로우인 `Assets/StreamingAssets/ComfyUI/car_video_workflow_api.json`을 사용한다.

### 이미지 생성 워크플로우

이미지 생성은 사용자의 그림을 단순 텍스트 프롬프트로만 생성하는 방식이 아니라, 사용자가 그린 이미지를 입력으로 넣는 img2img + ControlNet 방식이다.

1. Unity 캔버스에서 선 레이어와 색 레이어를 PNG로 추출한다.
2. 색 레이어는 img2img의 기준 이미지로 사용한다.
3. 선 레이어는 ControlNet Scribble 입력으로 사용해 자동차의 형태를 유지한다.
4. 선택한 스타일에 따라 프롬프트, 네거티브 프롬프트, denoise, 체크포인트, LoRA, ControlNet 강도를 바꾼다.
5. ComfyUI가 결과 이미지를 생성하면 Unity가 다운로드해 결과 화면에 표시한다.
6. 픽셀아트 스타일은 생성 후 추가 후처리로 픽셀 느낌을 정리한다.

### 스타일별 생성 방식

| 스타일 | 적용 방식 | 주요 내용 |
| --- | --- | --- |
| 실사 | 기본 SD 1.5 체크포인트 + img2img + ControlNet Scribble | 사용자의 색 레이어를 기준 이미지로 사용하고, 선 레이어를 ControlNet으로 넣어 자동차 형태를 유지한다. 프롬프트는 실제 자동차 사진 느낌, 금속 질감, 선명한 디테일을 강조한다. |
| 카툰 | 전용 체크포인트 방식 | 기본 실사 모델 대신 `toonyou_beta6.safetensors` 체크포인트를 사용한다. 카툰 윤곽선, 셀 셰이딩, 선명한 색감을 강조하며, 사람·캐릭터·얼굴이 붙은 자동차가 나오지 않도록 네거티브 프롬프트를 강화했다. |
| 픽셀아트 | LoRA 적용 방식 | 기본 모델 위에 `PixelArtRedmond15V.safetensors` LoRA를 적용하고, `Pixel Art, PixArFK` 트리거를 프롬프트 앞에 붙인다. 생성 후 `pixelateWidth`와 `pixelateColorLevels` 설정으로 픽셀아트 후처리를 적용한다. |
| 클레이 | 프롬프트 스타일링 방식 | 별도 체크포인트나 LoRA 없이 기본 모델을 사용하되, 클레이메이션·플라스틱 질감·둥근 형태를 프롬프트로 유도한다. 손, 사람, 받침대, 실제 금속 자동차 느낌은 네거티브 프롬프트로 억제한다. |

### 영상 생성 워크플로우

영상 생성은 이미지 결과를 먼저 사용자에게 보여준 뒤 백그라운드에서 진행한다.

1. 이미지 생성 결과 PNG와 선 레이어 PNG를 ComfyUI에 업로드한다.
2. `car_video_workflow_api.json` 워크플로우를 사용해 AnimateDiff 기반 mp4 생성을 요청한다.
3. 생성이 완료되면 Unity 결과 화면의 정지 이미지를 반복 재생 영상으로 교체한다.
4. 영상 생성 실패 또는 시간 초과 시에는 기존 이미지 결과를 그대로 유지한다.

이 구조 덕분에 영상 생성이 오래 걸리거나 실패해도 사용자는 결과 이미지를 즉시 확인할 수 있다.

## 3. 구동을 위한 최소 사양

### 필수 환경

- OS: Windows
- Unity: 2022.3.62f3 LTS
- CPU: 최신 데스크톱급 CPU 권장
- RAM: 최소 16GB, 안정적인 전시 운영 및 영상 생성 시 32GB 권장
- GPU: NVIDIA 8GB VRAM 이상 권장
  - 현재 검증 GPU: RTX 3060 8GB
- Python: 3.12.x
- 디스크: ComfyUI와 모델 포함 최소 10GB 이상 여유 공간 필요

### 필수 외부 구성

- 로컬 ComfyUI 서버 실행 필요
- 서버 주소: `http://127.0.0.1:8188`
- 기본 모델 파일:
  - SD 1.5 체크포인트
  - ControlNet Scribble
  - VAE
- 스타일별 추가 모델:
  - 카툰: ToonYou 체크포인트
  - 픽셀아트: PixelArtRedmond LoRA
- 영상 생성 사용 시 AnimateDiff 관련 노드와 모델이 추가로 필요하다.

## 4. 현재 운영 상태

- 기본 체험 흐름은 로컬 ComfyUI 기반으로 동작한다.
- 결과 이미지는 세션별로 저장되며, 설정이 준비된 경우 QR 업로드와 갤러리 등록까지 이어진다.
- 영상 생성은 활성화되어 있으며, 결과 화면에서 먼저 이미지를 보여준 뒤 mp4 생성을 백그라운드로 진행한다.
- 영상 생성 실패 또는 시간 초과가 발생해도 사용자는 이미지 결과를 계속 볼 수 있다.
- Backblaze B2 또는 GCS 설정이 없으면 QR 기능은 자동으로 숨겨진다.
- 콘텐츠 필터는 현재 기본 설정상 비활성화되어 있어, 전시 운영 기준에 따라 활성화 여부를 결정해야 한다.

---

## 5. 운영/보고 시 확인 항목

### 1) 서버 및 생성 상태

- ComfyUI 서버 정상 기동 여부
- 생성 시간, 성공률, 실패 원인
- 첫 생성 또는 스타일 전환 시 모델 로딩 지연 여부
- 서버 재시작 후 정상 복구 여부

### 2) 품질 상태

- 스타일별 결과 품질: 실사, 카툰, 픽셀아트, 클레이가 의도한 분위기로 나오는지 확인
- 형태 유지: 차체, 바퀴 4개, 창문, 전체 자동차 실루엣 유지 여부
- 색상 유지: 사용자가 칠한 색이 과도하게 변형되지 않는지 확인
- 취약 입력: 화면 구석에 작게 그린 그림, 반쯤 잘린 그림, 낙서성 입력의 품질 저하 여부
- 픽셀아트는 LoRA와 후처리 설정을 사용하므로 이미지와 영상 모두 같은 스타일로 유지되는지 확인

### 3) 외부 연동 상태

- Backblaze B2 또는 GCS 업로드 성공 여부
- QR 링크 생성 및 모바일 다운로드 가능 여부
- 갤러리 등록 버튼 동작 여부
- Display 2 갤러리 반영 여부
- 콘텐츠 필터를 켤 경우 Gallery/Quarantine 분기 정상 동작 여부

### 4) 운영 리스크

- ComfyUI 서버가 꺼져 있거나 모델이 누락되면 생성 실패가 발생한다.
- 워크플로우 JSON의 노드 ID가 바뀌면 Unity 쪽 치환 코드도 함께 수정해야 한다.
- 영상 생성은 이미지 생성보다 시간이 길며, 전시 PC 성능에 따라 대기 시간이 달라질 수 있다.
- B2/GCS 키 파일은 저장소에 포함하지 않고 운영 PC의 `Config/` 폴더에 별도 배치해야 한다.
- 모델, 워크플로우, 체크포인트 변경 후에는 반드시 회귀 검증이 필요하다.

---

## 6. 남은 결정 사항

- B2 계정과 버킷을 실제 운영용으로 개통할지 결정
- QR 링크 유효기간을 현재 7일 기준으로 유지할지 결정
- 콘텐츠 필터를 켤지, 관리자 사후 검수 방식으로 운영할지 결정
- 갤러리에 영상까지 포함할지, 이미지만 전시할지 결정
- 관리자 모드, 빌드 세팅, 전시 PC 자동 실행 방식 검증
- 실제 전시 PC의 CPU, RAM, GPU, 저장 공간 최종 확인

---

## 7. 한 줄 결론

현재 시스템은 "그리기 → 생성 → 결과 표시 → 저장/업로드/전시" 흐름까지 동작하며, 운영 보고는 서버 상태, 생성 품질, QR/갤러리 연동, 콘텐츠 필터 운영 방식, 영상 생성 안정성, 전시 PC 사양 확인을 중심으로 정리하면 된다.
