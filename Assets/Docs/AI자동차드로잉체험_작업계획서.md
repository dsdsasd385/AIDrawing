# 1. 프로젝트 개요

### 프로젝트명

AI 자동차 드로잉 체험 (AI Car Drawing Experience)

### 개발 목적

전시장 설치형 인터랙티브 체험물. 관람객이 마우스로 자동차를 자유롭게 그리면,
로컬 AI(Stable Diffusion)가 그 낙서를 완성도 높은 자동차 이미지로 변환해 보여준다.

게임(점수/승패)이 아니라 **창작 체험 + 전시 볼거리**가 목적이다.
관람객의 작품은 본인 선택(opt-in)에 따라 갤러리 슬라이드쇼에 전시되고,
QR 코드로 자신의 폰에 다운로드할 수 있다.

### 완성 기준

**전시 배포 수준** — 부팅 자동 시작, 생성 서버 워치독, 키오스크 전체화면, 관리자 도구까지 포함.
프로토타입이 아니라 실제 전시장에 놓을 수 있는 상태가 목표.

# 2. 개발 환경

| 항목        | 내용                                             |
| --------- | ---------------------------------------------- |
| Engine    | Unity 2022.3.62f3 LTS                          |
| 개발/운영 플랫폼 | Windows                                        |
| GPU       | NVIDIA RTX 3060 **8GB** (실측 확인, 12GB 아님)       |
| 조작 방식     | 마우스                                            |
| UI 방식     | 2D UI (uGUI)                                   |
| 해상도       | 1920×1080 (키오스크), Display 2 갤러리 지원             |
| AI 생성 서버  | ComfyUI (API 모드, 로컬 HTTP)                      |
| 생성 모델     | Stable Diffusion 1.5 + ControlNet Scribble     |
| 생성 목표 시간  | 4~8초                                           |
| QR 업로드    | Backblaze B2 (무료 티어, 공개 버킷 — GCS는 대안)          |
| 자동 필터     | 경량 로컬 VLM (CPU, 비동기)                           |

### VRAM 8GB 제약에서 나온 결정 (변경 금지 근거)

* SDXL 계열은 8GB에서 오프로딩 필요 → 10~20초 + OOM 리스크 → **제외**
* SD 1.5 + ControlNet은 8GB에 상주 가능 → **채택**
* 필터용 VLM은 GPU를 SD가 점유하므로 **CPU에서 비동기 실행**

# 3. 시스템 구성도

```
[Unity 앱 (키오스크)]
   │  스케치 PNG 2장 업로드 + 워크플로 JSON 제출 (HTTP)
   ▼
[ComfyUI 서버 (localhost:8188)]
   │  SD 1.5 + ControlNet Scribble + img2img
   ▼
[생성 이미지 PNG]
   ├──> Unity 결과 화면 표시 (즉시)
   ├──> 로컬 저장 (스케치+결과 쌍, 항상)
   ├──> GCS 업로드 → QR 표시 (비동기, 실패해도 체험 계속)
   └──> 관람객이 "전시하기" 선택 시
           → VLM 필터 검사 (CPU, 비동기)
           → 통과 시 갤러리 폴더 → Display 2 슬라이드쇼
```

핵심 원칙: **생성(핵심 체험)은 인터넷 없이 동작**한다.
QR 업로드는 부가 기능이며, 인터넷 단절 시 QR 버튼만 자동으로 숨긴다.

# 6. 그리기 캔버스 설계

### 이중 텍스처 (핵심 설계)

캔버스는 **두 장의 RenderTexture**에 동시에 그린다.

| 레이어        | 내용            | AI에서의 용도                     |
| ---------- | ------------- | ---------------------------- |
| LineLayer  | 검정 선만 (색 무시)  | ControlNet Scribble 입력 (형태)  |
| ColorLayer | 색 포함 전체 그림    | img2img 초기 이미지 (색상)          |

이렇게 하면 "관람객이 칠한 색"과 "그린 형태"가 각각 제 역할로 생성에 반영된다.

### 캔버스 사양

* 해상도: **768×512 (가로형, 자동차 비율)** — SD 1.5 안정 범위
* 배경: 흰색

### 그리기 도구

* 펜 (굵기 3단계)
* 색상 팔레트 (고정 색상 8~12개)
* 지우개
* 되돌리기 (undo, 스트로크 단위)
* 전체 지우기

### 구현 방식

* 마우스 드래그 → 두 RenderTexture에 선분 스탬핑 (Graphics.Blit + 브러시 셰이더 또는 GL 라인)
* undo는 스트로크 시작 시점의 텍스처 스냅샷 스택 (최대 N개 제한)
* 완성 시 `Texture2D.ReadPixels` → PNG 인코딩 2장

# 7. AI 생성 파이프라인

### ComfyUI 서버

* 주소: `http://127.0.0.1:8188`
* 실행: API 모드 (헤드리스 가능), Windows 부팅 시 자동 시작 (12장 참조)

### 필요 모델 파일 (설치 시 준비물)

| 종류          | 파일                                          | 위치                                   |
| ----------- | ------------------------------------------- | ------------------------------------ |
| 체크포인트       | SD 1.5 계열 차량/실사 특화 커뮤니티 모델 (마일스톤 ①에서 선정)   | `ComfyUI/models/checkpoints/`        |
| ControlNet  | `control_v11p_sd15_scribble.pth`             | `ComfyUI/models/controlnet/`         |

### 워크플로 (JSON으로 고정 저장)

```
ColorLayer PNG ──> VAE Encode ──────────────┐
                                            ▼
LineLayer PNG ──> ControlNet Scribble ──> KSampler (img2img, denoise=스타일값)
                                            ▼
프롬프트(스타일값) ─────────────────────> VAE Decode ──> 결과 PNG
```

### Unity ↔ ComfyUI 연동 (UnityWebRequest)

1. `POST /upload/image` — LineLayer, ColorLayer PNG 업로드
2. `POST /prompt` — 워크플로 JSON 제출 (이미지 파일명 + 스타일 프롬프트 + 디노이즈 치환)
3. `GET /history/{prompt_id}` 폴링 (0.5초 간격) — 완료 확인
4. `GET /view?filename=...` — 결과 PNG 다운로드

### 튜닝 핵심 변수

* **디노이즈 강도**: 낮으면 낙서 티가 남고, 높으면 관람객의 색이 날아감.
  스타일 프리셋마다 (프롬프트 + 디노이즈 값) 쌍으로 저장하고 개별 튜닝한다.

# 8. 스타일 프리셋

* 스타일 선택 화면은 **4종 슬롯**으로 구성: 실사 / 카툰 / 픽셀아트 / 클레이 (클레이의 id는 `simplecolor` — 초기 「채색」 스타일에서 개명)
  * **4종 모두 프롬프트 확정·API 생성 검증 완료 (2026-07-10)**. 공통 구조: 단일 완성차 가중치 앵커 + 바퀴 접지 + 스타일 키워드 + 기형 억제 부정 프롬프트. denoise는 스타일별: 실사·클레이 0.8(모양 우선), 카툰·픽셀 0.7(색 보존 우선 — 시점별 매트릭스 검증) (인수인계 §5)
* 스타일은 코드가 아니라 **JSON 데이터**로 관리 (운영자 수정 가능). 각 스타일은 `thumbnail`에 예시 이미지 경로(StreamingAssets 기준)를 가진다
* 스타일별 **선택 필드** (생략하면 기본 동작, 2026-07-13 신설):
  * `checkpoint` — 전용 체크포인트 파일명. 실사 모델(워크플로 기본)로는 화풍이 안 나오는 스타일만 지정. 현재 카툰=`toonyou_beta6.safetensors` (실사 모델로는 카툰 품질 상한이 낮음). 스타일이 바뀌는 첫 생성은 모델 스왑으로 몇 초 지연
  * `controlnetStrength` — ControlNet 강도 (0=워크플로 기본 0.35). 카툰 0.5 — ToonYou는 상상력이 강해 약하게 걸면 스케치 구도를 무시(정면 뷰 등으로 재해석)
  * `lora` / `loraStrength` / `loraTrigger` — 체크포인트 교체 없이 화풍만 얹는 LoRA. 현재 픽셀아트=`PixelArtRedmond15V.safetensors`(강도 0.8, 트리거 `Pixel Art, PixArFK`). **이미지·영상 양쪽에 주입**된다 — 영상도 같은 화풍을 유지해야 하기 때문(인수인계 §6). 실사 모델+프롬프트만으로는 픽셀 품질 상한이 낮고, 강한 후처리 픽셀화는 모자이크가 됨
  * `pixelateWidth` / `pixelateColorLevels` — 결과 PNG 후처리(PixelArtFilter)·영상 재픽셀화: 블록 평균 축소 + 니어리스트 확대로 픽셀 그리드를 균일하게 스냅. 픽셀아트 160 / 8 (LoRA가 디테일을 만드니 그리드 정리만; 값이 작을수록 거칠어짐). 0이면 후처리 없음
* 화면 구성은 씬에 배치한 VerticalGroup 4개(각 예시 이미지 + 버튼)를 `StylePanelController.styleGroups`에 순서대로 연결. Styles.json 순서 = 슬롯 순서, 스타일 수보다 많은 슬롯은 자동 숨김

```json
{
  "styles": [
    {
      "id": "realistic",
      "name": "실사",
      "prompt": "a photorealistic car, studio lighting, ...",
      "negativePrompt": "...",
      "denoise": 0.7,
      "thumbnail": "StyleExamples/realistic.png"
    }
  ]
}
```

`thumbnail`은 스타일 버튼에 보여줄 **예시 결과 이미지**의 경로(StreamingAssets 기준)다. 파일이 없으면 씬에 배치한 임시 이미지를 그대로 둔다.

# 9. 결과 처리

체험 1회의 산출물: 스케치 PNG + 결과 PNG 쌍. 처리 경로 3가지:

### 9-1. 화면 표시 (항상)

결과 화면에서 스케치 vs 완성본 비교 표시.

### 9-2. QR 다운로드 (인터넷 필요, 자동 숨김)

* 생성 직후 결과 PNG를 **Backblaze B2 비공개 버킷**에 비동기 업로드 (2026-07-10 GCS에서 전환 —
  카드 등록 없이 10GB 영구 무료 + 다운로드는 평균 저장량 3배까지 무료, 전시 볼륨에 충분)
* 버킷을 비공개로 두는 이유: B2는 **공개 버킷에만 카드 등록을 요구**한다. 대신 QR 링크에
  만료형 다운로드 토큰을 붙인다 — **링크 유효 7일**(B2 최대치). 관람객은 현장에서 스캔하므로 충분
* QR은 PNG 직링크가 아니라 **[이미지 저장하기] 버튼이 있는 모바일 랜딩 페이지**를 가리킨다 —
  결과 PNG와 랜딩 HTML을 같은 접두사로 업로드하고, 다운로드 토큰 1개(fileNamePrefix)로 둘 다 커버.
  랜딩 템플릿은 `StreamingAssets/Data/QrLanding.html` (문구·디자인 운영자 수정 가능, 13장 원칙).
  HTML 업로드가 실패하면 PNG 직링크로 자동 폴백해 QR은 계속 동작한다
* QR 내용: `<downloadUrl>/file/<버킷>/<파일명>.html?Authorization=<토큰>`
* 애플리케이션 키는 **해당 버킷 제한 + writeFiles·shareFiles 권한만** 부여 (전시장 PC 유출 대비 최소 권한)
* 업로드 실패/오프라인 → QR 영역만 숨기고 체험은 정상 진행
* 업로더는 인터페이스(`IResultUploader`) 뒤에 두어 스토리지 교체 가능 — GCS 구현(`GcsUploader`)도
  대안으로 유지하며, B2 미설정 시 GCS가 설정돼 있으면 자동으로 그쪽을 쓴다 (기본 B2 → 폴백 GCS)

### 9-3. 갤러리 전시 (관람객 opt-in + 자동 필터)

* 결과 화면의 [전시장에 내 작품 걸기] 버튼을 누른 작품만 후보가 된다
* 후보는 VLM 필터(10장) 통과 후 갤러리 폴더에 복사 → Display 2 슬라이드쇼에 반영

### 로컬 저장 구조 (모든 체험 기록)

```
<persistentDataPath>/
├── Sessions/          ← 전체 기록 (스케치+결과 쌍, 항상 저장)
│   └── 20260708_143012_sketch.png / _result.png
├── Gallery/           ← 필터 통과 + opt-in 작품만 (슬라이드쇼가 이 폴더를 감시)
├── Quarantine/        ← 필터에 걸린 opt-in 작품 (관리자가 복원 가능)
└── Logs/
```

# 10. 자동 필터 (갤러리 게이트)

### 방식

* **경량 로컬 VLM** (Moondream2 또는 Qwen2-VL-2B급)에게 스케치와 결과 이미지를 보여주고 질문:
  "차량 그림인가? 성적·혐오·부적절 문자 낙서가 포함되는가?"
* **CPU에서 비동기 실행** (2~5초) — 갤러리 등재만 결정하므로 관람객은 기다리지 않는다
* 사진 학습 NSFW 분류기를 쓰지 않는 이유: 낙서/선화에 잘 동작하지 않음. 최대 리스크는 스케치 쪽
* 호출 방식: **OpenAI 호환 chat completions API** (Ollama·llama.cpp 서버 등) — 모델을 바꿔도 코드 수정 없이
  Config.json `filter.model`/`endpoint`만 교체. 판정 질문도 `filter.question`으로 데이터화 (13장 원칙)
* 응답 계약: VLM은 `{"ok": true/false}` JSON 한 줄로 답하게 강제. 해석 실패·타임아웃·서버 부재 = 판정 불가 → 격리

### 판정 정책

* **보수적**: 애매하면 갤러리 제외 (본인 화면·QR은 정상 제공되므로 피해 없음)
* 제외된 작품은 Quarantine 폴더로 → 관리자가 복원 가능
* 등재된 작품도 관리자가 즉시 삭제 가능 (양방향 교정)

# 17. 코드 주석 작성 규칙 (기존 규칙 승계)

* 모든 스크립트는 **한국어 주석**으로 작성한다
* 주석은 "무엇"이 아니라 **"왜"와 "역할"**을 설명한다
* 클래스 상단: `/// <summary>`로 역할, 소속 시스템, 다른 클래스와의 관계
* `public` 메서드: 기능, `<param>`, `<returns>`
* `public` / `[SerializeField]` 필드: 용도를 `/// <summary>`로
* 조건 분기·계산식·예외 처리 등 의도가 안 드러나는 코드에는 한 줄 주석으로 이유 명시
* 튜닝 수치(디노이즈, 시간 정책 등)는 근거를 함께 명시
